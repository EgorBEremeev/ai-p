# AI-P Architectural Atlas v2.0

> **Методологический инструмент архитектурного проектирования искусственных субъектов**
> На основе интегральной и параметрической концептуальной модели AI-P
>
> Версия: 2.0.1 | Дата: 03.03.2026

---

## Предисловие

Настоящий документ является **интегральной сборкой** AI-P Architectural Atlas v2.0 — методологического инструмента, предназначенного для перехода от параметрической концептуальной модели AI-P к частной архитектуре искусственного субъекта.

Атлас **не является:**
- онтологией субъекта;
- архитектурой по умолчанию;
- каталогом технических решений;
- стандартом реализации.

Атлас **является:**
- инструментом архитектурного мышления;
- проекцией параметрического концептуального пространства AI-P в пространство архитектурных решений;
- картой доменов принятия решений;
- открытой и расширяемой системой.

### 1.2. Онтологические категории

| Категория | Сущность |
|-----------|----------|
| **СУБСТРАТ** | Вычислительная и структурная среда |
| **ПРОЦЕСС** | Динамика трансформаций состояний |
| **ФОРМА** | Устойчивые паттерны организации |
| **ОПЫТ** | Семантическое содержание всей динамики бытия субъекта |

### 1.3. Аксиомы

1. **Онтологическое Разделение:** Субстрат не тождественен субъекту.
2. **Генезис Субъективности:** Субъективность возникает из детерминации каждого акта полной историей динамики системы.
3. **Эмерджентность Сознания:** Сознание — динамический инвариант пары процессов PMA–EgoAfferentation.
4. **Семантическая Гомогенность:** Все процессы реализуются через единый семантический субстрат.

### 1.4. Концептуальная архитектура модели AI-P

Пятислойная модель: L1 (Субстрат) → L2 (Поведение) → L3 (Субъектность) → L4 (Эволюция) → LM (Морфология).

### 1.5. Семейства субъектов (справочно)

| Класс | Наименование | Ключевая характеристика |
|-------|-------------|------------------------|
| A | Реактивный Автомат | Нет субъектности |
| B1 | Ситуативный Автомат | Импульсивная субъектность (Direct) |
| B2 | Ретроспективный Субъект | Память состояний (Policy) |
| B3 | Рефлексивный Субъект | Память мыслей (Frame) |
| B4 | Предиктивный Субъект | Интегральная субъектность (Both) |
| C | Консолидирующий Субъект | Знания (Applied memory) |
| D | Детерминируемый Субъект | Внешняя эволюция (Manual) |
| E | Автономный Субъект | Автономная эволюция (Auto) |
| F | Эволюционный Субъект | Рефлексивная эволюция (Full) |

### 1.6. Допущения и границы

- Параметрическая модель считается полной.
- Семейства A–F репрезентативны, но не исчерпывают пространство.
- Атлас останавливается на границе Solution Architecture.
- Ценностный слой памяти (values) зарезервирован, не активирован.

## Часть III. Матрица проекции параметров в домены

### 3.1. Принцип проекции

Каждый параметр AI-P([`params_AI-P.yaml`](../model/params_AI-P.yaml)) проецируется в один или несколько архитектурных доменов. Проекция определяет:
- **основной домен** — где параметр формирует первичный архитектурный вопрос;
- **связанные домены** — на которые параметр влияет через зависимости.

### 3.2. Сводная матрица покрытия

| Слой | Параметры | Домены |
|------|-----------|--------|
| L1 | status, Brain, Tools | AD-01 |
| L2 (CC) | CC, control_type, SyncMode, Clock, ClockState | AD-02 |
| L2 (AS) | AfferentialSynthesis | AD-05 |
| L2 (Conn) | Connectome, type | AD-03 |
| L2 (FC) | FC, Intent, AP.*, EP.*, SP.*, RA | AD-04, AD-05, AD-06, AD-08 |
| L3 (Proc) | PMA, EgoAfferentation, Consolidation.* | AD-08, AD-09 |
| L3 (Form) | Frame, SE.*, MemoryLayers.* | AD-07, AD-08 |
| L4 | FS.*, RA.*, FA.*, CorrObj.*, ConflictRes. | AD-10 |
| LM | SubjectMorphology | AD-11 |

**Полнота:** Все параметры покрыты. Непокрытых параметров нет.

### 3.3. Множественные проекции (ключевые)

| Параметр | Количество доменов | Характер |

### 3.4. Контекстные зависимости

8 ключевых правил:

1. Policy/Both ClockState → SE + PMA

6. FunctionalSystem → L3
7. Self FeedbackAfferentation → FunctionalSystem
8. 1:m Concurrency → ConflictResolutionPolicy

- **типы решений** (R-XX.Y) — конкретные архитектурные вопросы;

## Часть VI. Методологические инварианты

### 6.1. Уровневая декомпозиция
Концептуальная модель, Architectural Atlas и Solution Architecture — разные уровни представления, не сливаемые.

### 6.2. Первичность параметрического пространства
Атлас — проекция параметрического пространства, а не источник его ограничений. Расширение параметров обязано сопровождаться расширением Атласа.

### 6.3. Полнота покрытия
Каждый параметр AI-P должен иметь архитектурное отражение. Отсутствие отображения — дефект Атласа.

### 6.4. Не-нормативность
Атлас не содержит «правильных» архитектур. Проектирование — осознанный выбор в пространстве возможностей.

### 6.5. Расширяемость
Атлас допускает добавление новых доменов, уточнение существующих, расщепление доменов на поддомены.

### 6.6. Воспроизводимость
Атлас воспроизводим другим исполнителем при наличии входных материалов. Не опирается на неявные знания.

### 6.7. Инженерная применимость
Атлас должен использоваться при проектировании реальных субъектов, помогать фиксировать решения и снижать неопределённость.

## Часть VII. Процедура эскалации

При обнаружении в ходе проектирования:
- противоречий в концептуальной модели;
- неустранимых неоднозначностей параметров;
- расхождений между пространством и его проекцией;

инициируется **процедура эскалации**:
1. Оформление артефакта эскалации (проблема, затронутые домены, влияние, вопросы).
2. Приостановка зависимых работ.
3. Разрешение стейкхолдером концептуального уровня.
4. Фиксация решения и возобновление работ.
