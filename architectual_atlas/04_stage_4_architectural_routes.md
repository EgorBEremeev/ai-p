# ЭТАП 4 — Описание проектного маршрута использования Architectural Atlas

> **AI-P Architectural Atlas** | Артефакт этапа 4
> Версия: 2.0 | Дата формирования: 03.03.2026

---

## 1. Назначение документа

Документ описывает **воспроизводимый маршрут** перехода от частной концептуальной модели AI-P к частной архитектуре субъекта с использованием Architectural Atlas.

**Важное методологическое замечание:** Архитектурный маршрут носит описательный и методологический характер. Он **не является** алгоритмом, формальной процедурой или исполняемым пайплайном, а служит средством навигации проектировщика по пространству архитектурных решений.

---

## 2. Предварительные условия

Для использования маршрута проектировщик должен располагать:

1. **Частной концептуальной моделью** — выбранная конфигурация параметров AI-P (заполненный шаблон `decision_table_TEMPLATE_AI-P.md` или эквивалент).
2. **Architectural Atlas** — настоящий набор артефактов (домены, матрица проекции, каталог решений).
3. **Понимание интегральной модели AI-P** — знание онтологических категорий, слоёв, аксиом.

Маршрут **не требует:**
- знания конкретных технологий или фреймворков;
- опыта реализации агентных систем;
- доступа к автору AI-P.

## 4. Детальное описание шагов

### ШАГ 1: Фиксация частной концептуальной модели

**Цель:** Однозначно зафиксировать параметрическую конфигурацию проектируемого субъекта.

**Действия проектировщика:**

1. Заполнить шаблон-конфигуратор (`decision_table_TEMPLATE_AI-P.md`), указав для каждого параметра конкретное значение из его ОДЗ.

2. Проверить соблюдение структурных зависимостей:
   - Frame → AfferentialSynthesis
   - Memory (PMA) → AfferentialSynthesis
   - Накопление опыта → EgoAfferentation
   - SelectionDepth → соответствующие слои памяти
   - Консолидация → соответствующие слои памяти
   - FunctionalSystem → L3 (SubjectiveExperience)
   - FeedbackAfferentation → FunctionalSystem
   - Connectome.Implicit → SubjectiveExperience
   - Адаптивный CC → Procedural Memory
   - AfferentialPolicy.ClockState: Policy → SE + PMA

3. Идентифицировать ближайшее семейство субъекта (A–F) для ориентации. Частная модель может не совпадать полностью ни с одним из 9 семейств — это допустимо.

**Результат шага:** Заполненная таблица решений (конфигурация параметров) с подтверждением соблюдения всех зависимостей.

**Точка архитектурного выбора:** Собственно, сама конфигурация параметров уже является первичным архитектурным выбором на концептуальном уровне. Проектировщик должен осознавать, что за каждым значением параметра стоит архитектурное следствие.

---

### ШАГ 2: Определение активных архитектурных доменов

**Цель:** Выяснить, в каких доменах проектировщику необходимо принять решения.

**Действия проектировщика:**

1. Использовать **матрицу проекции** (`projection_matrix.md`, раздел 5: «Проекция семейств субъектов в домены») для определения активных доменов.

2. Для каждого из 11 доменов определить **уровень активации:**

   | Уровень | Описание | Действие |
   |---------|----------|----------|
   | **Неактивен** | Домен не задействован данной конфигурацией | Пропустить |
   | **Базовый** | Домен активен в минимальной конфигурации | Принять базовые решения |
   | **Расширенный** | Домен активен с дополнительными возможностями | Принять решения по всем осям, включая    расширения |
   | **Полный** | Все возможности домена задействованы | Принять решения по всем осям вариативности |

3. Построить **профиль активации доменов** — таблицу, показывающую уровень активации для каждого из 11 доменов.

**Результат шага:** Профиль активации доменов — список доменов с указанием уровня активации.

**Пример профиля** (для модели класса C — Консолидирующий Субъект):

| Домен | Уровень | Обоснование |
|-------|---------|-------------|
| AD-01 | Расширенный | Brain + Tools |
| AD-02 | Расширенный | +Async, Стохастический |
| AD-03 | Базовый | Implicit |
| AD-04 | Полный | Все политики активны |
| AD-05 | Расширенный | +PMA(Applied) |
| AD-06 | Расширенный | +Tools, SubjectiveContext=Both |
| AD-07 | Расширенный | Ep(Unlim) + Applied |
| AD-08 | Полный | PMA + EgoAff + Frame |
| AD-09 | Базовый | Ep + Applied консолидация |
| AD-10 | Неактивен | L4 = Absent |
| AD-11 | Базовый | Carrier без модификации |

### ШАГ 3: Навигация по доменам — принятие архитектурных решений

**Цель:** Для каждого активного домена принять осознанные архитектурные решения.

**Действия проектировщика:**

1. **Последовательность навигации.** Рекомендуемый (но не обязательный) порядок обхода доменов:

```
AD-01 (Субстрат) → AD-11 (Морфология) →
AD-02 (Управление) → AD-03 (Коннектом) → AD-04 (ФЯ) →
AD-05 (Афференция) → AD-06 (Эфференция) →
AD-07 (Опыт) → AD-08 (Проекция) →
AD-09 (Консолидация) → AD-10 (Эволюция)
```

Логика порядка:
- Сначала инфраструктура (где и чем: AD-01, AD-11).
- Затем поведенческий каркас (как управлять, как связаны, как устроен акт: AD-02, AD-03, AD-04).
- Затем триада когнитивного акта (контекст и действие: AD-05, AD-06).
- Затем субъектность (опыт и сознание: AD-07, AD-08).
- Затем развитие (обучение и эволюция: AD-09, AD-10).

2. **Для каждого активного домена:**

   a. Открыть описание домена в `architectural_domains.md`.
   b. Открыть соответствующий раздел в `solution_space_catalog.md`.
   c. Для каждой **оси архитектурного выбора** в домене:
      - Определить текущее сужение пространства решений (на основе значений параметров).
      - Рассмотреть диапазон вариативности, не сужаемый параметрами (архитектурная свобода).
      - Принять решение или зафиксировать открытый вопрос.
   d. Осознать **архитектурные напряжения**, перечисленные в каталоге.
   e. Зафиксировать принятые решения в структурированной форме.

3. **Формат фиксации решения по домену:**

```markdown
### [ID Домена]: [Название]

**Уровень активации:** [Неактивен / Базовый / Расширенный / Полный]

**Принятые решения:**
| Решение | Выбор | Обоснование |
|---------|-------|-------------|
| R-XX.Y  | ...   | ...         |

**Осознанные напряжения:**
- [описание напряжения и принятого компромисса]

**Открытые вопросы:**
- [если есть — вопросы, требующие уточнения на уровне Solution Architecture]
```

**Результат шага:** Набор решений по каждому активному домену.

**Точки архитектурного выбора:** Каждая ось вариативности в каждом домене — точка выбора. Проектировщик фиксирует: *что выбрано, почему, и какие компромиссы приняты.*

---

### ШАГ 4: Проверка согласованности решений между доменами

**Цель:** Убедиться, что решения, принятые в разных доменах, не противоречат друг другу.

**Действия проектировщика:**

1. **Проверка контекстных зависимостей** (из `projection_matrix.md`, раздел 4):
   - Для каждой зависимости убедиться, что решения в связанных доменах согласованы.
   - Пример: если в AD-08 выбрана глубокая проекция (SelectionDepth = Applied), в AD-07 должен быть спроектирован прикладной слой памяти, а в AD-09 — механизм AppliedConsolidation.

2. **Проверка межсменных напряжений** (из `solution_space_catalog.md`, раздел 3):
   - Для каждого из 8 напряжений между доменами убедиться, что принятые решения образуют согласованный компромисс.
   - Пример: напряжение «Афференция vs. Эфференция» (AD-05 ↔ AD-06) — мощность контекста в AD-05 должна быть сбалансирована с возможностями трансляции в AD-06.

3. **Проверка взаимосвязей доменов** (из `architectural_domains.md`, раздел 5: схема взаимосвязей):
   - Для каждой связи между доменами убедиться, что решения на обоих концах связи совместимы.

4. **Фиксация несогласованностей:**
   - Если обнаружена несогласованность → вернуться к шагу 3 для корректировки.
   - Если несогласованность неустранима → зафиксировать как архитектурное ограничение.

**Результат шага:** Подтверждение согласованности или перечень корректировок.

---

### ШАГ 5: Фиксация трассируемости

**Цель:** Обеспечить прослеживаемость каждого архитектурного решения до исходных параметров модели.

**Действия проектировщика:**

1. Для каждого принятого решения (R-xx.y) построить цепочку трассируемости:

```
параметр AI-P → домен (AD-xx) → решение (R-xx.y) → принятый компромисс
```

2. Проверить отсутствие «висящих» решений — решений, не прослеживаемых до параметров. Если такое решение обнаружено, оно либо ошибочно (не нужно), либо указывает на пробел в матрице проекции.

3. Зафиксировать трассировочную матрицу.

**Формат трассировочной записи:**

```markdown
| Параметр | Домен | Решение | Выбор | Компромисс |
|----------|-------|---------|-------|------------|
| AP.ClockState=Policy | AD-05 | R-05.4 | SelectionDepth=Episodic | Глубина vs. Латентность |
```

**Результат шага:** Трассировочная матрица проектного выбора.

---

### ШАГ 6: Проверка критических инвариантов

**Цель:** Убедиться, что методологические инварианты Атласа соблюдены.

**Действия проектировщика:**

Проверить каждый инвариант:

| # | Инвариант | Проверочный вопрос | Статус |
|---|-----------|-------------------|--------|
| 1 | Уровневая декомпозиция | Концептуальная модель, Atlas и Solution Architecture — различные уровни, не слитые? | ☐ |
| 2 | Первичность параметрического пространства | Atlas — проекция параметрического пространства, а не источник его ограничений? | ☐ |
| 3 | Полнота покрытия | Каждый параметр конфигурации имеет отражение в доменах? | ☐ |
| 4 | Не-нормативность | Решения по осям — результат выбора проектировщика, а не предписание Атласа? | ☐ |
| 5 | Расширяемость | Принятые решения не блокируют возможность расширения Атласа? | ☐ |
| 6 | Воспроизводимость | Другой проектировщик с теми же входными данными получит тот же набор доменов и осей? | ☐ |
| 7 | Инженерная применимость | Результат пригоден для передачи в Solution Architecture? | ☐ |

**Результат шага:** Заполненный чек-лист инвариантов.

---

### ШАГ 7: Контроль границы Solution Architecture

**Цель:** Убедиться, что частная архитектура не содержит реализационных деталей.

**Действия проектировщика:**

1. Проверить отсутствие в артефактах:
   - Упоминаний конкретных технологий, библиотек, SDK.
   - Описаний API-контрактов или форматов данных.
   - Оптимизационных решений (производительность, масштабирование).
   - Рекомендаций по конкретным инструментам.

2. Если обнаружены элементы Solution-уровня — перенести их в раздел «Открытые вопросы для Solution Architecture».

**Индикаторы нарушения границы:**
- Упоминание конкретных LLM, баз данных, фреймворков.
- Описание протоколов взаимодействия.
- Количественные ограничения (размер памяти, задержка, throughput).
- Схемы развёртывания или инфраструктурные решения.

**Результат шага:** Подтверждение соблюдения границы или перечень элементов, перенесённых в открытые вопросы.

## 6. Работа с эволюционными линиями

### 6.1. Проектирование эволюционирующего субъекта

Если проектируемый субъект предполагает **эволюцию** (переход между семействами), маршрут дополняется:

1. Зафиксировать **начальное состояние** (стартовый класс субъекта).
2. Зафиксировать **целевое состояние** (целевой класс).
3. Определить **эволюционные переходы** между ними (из `projection_matrix.md`, раздел 5.2).
4. Для каждого перехода определить:
   - какие домены активируются / усложняются;
   - какие новые решения необходимо принять;
   - какие существующие решения должны быть совместимы с будущим расширением.

### 6.2. Проектирование «с запасом» (forward compatibility)

При проектировании субъекта начального класса (например, B3), который в будущем может эволюционировать (до C, D), проектировщик может:

- В AD-07: заложить расширяемую структуру памяти (подготовить к появлению Applied слоя).
- В AD-11: обеспечить поддержку модификации структур «на лету» (для будущего L4).
- В AD-04: спроектировать политики ФЯ как пластичные (для будущей эволюции).
- В AD-06: предусмотреть расширение TargetSubstrate до Both (для будущего добавления Tools).

Это не обязательно, но повышает эволюционную готовность архитектуры.

## 7. Итерации в маршруте

Маршрут допускает возвраты и итерации:

- **Шаг 3 → Шаг 2:** Если при навигации по домену обнаружилось, что уровень активации определён неверно.
- **Шаг 4 → Шаг 3:** Если проверка согласованности выявила конфликт.
- **Шаг 5 → Шаг 3:** Если при фиксации трассируемости обнаружено «висящее» решение.
- **Шаг 6 → Шаг 3:** Если проверка инвариантов выявила нарушение.
- **Шаг 7 → Шаг 3:** Если обнаружено пересечение границы Solution Architecture.
- **Шаг 3 (домен X) → Шаг 3 (домен Y):** Если решение в одном домене влияет на уже принятое решение в другом.

Итерации допустимы и ожидаемы. Маршрут — не линейный конвейер, а **навигационный инструмент** проектировщика.
