# ЭТАП 3 — Каталог пространства архитектурных решений по доменам

> **AI-P Architectural Atlas** | Артефакт этапа 3
> Версия: 2.0 | Дата формирования: 03.03.2026

---

## 1. Назначение каталога

Каталог фиксирует **пространство возможных архитектурных решений** внутри каждого домена AI-P Architectural Atlas. Для каждого домена описаны:

- типы архитектурных решений, которые необходимо принять;
- диапазоны вариативности (спектр возможных ответов);
- архитектурные компромиссы и напряжения;
- влияние параметров AI-P на сужение пространства решений.

**Принципиальное ограничение:** Каталог описывает *пространство возможностей*, а не рекомендуемые решения. Он не содержит «правильных» архитектур, оптимальных конфигураций или целевых состояний.

## 2. Каталог по доменам

### AD-01: Когнитивный субстрат

#### Типы архитектурных решений

| Решение | Диапазон вариативности | Управляющие параметры |
|---------|----------------------|----------------------|
| **R-01.1** Природа семантического процессора | Единый монолитный процессор ←→ Федерация специализированных процессоров | `Brain` |
| **R-01.2** Модальности сенсорных интерфейсов | Отсутствие ←→ Единственная модальность ←→ Мультимодальность | `Tools` |
| **R-01.3** Модальности моторных интерфейсов | Отсутствие ←→ Генерация текста/кода ←→ Физические действия | `Tools` |
| **R-01.4** Разграничение когнитивных и моторных актов | Единый субстрат для обоих ←→ Разделённые субстраты | `Brain`, `Tools`, `EfferentialPolicy.TargetSubstrate` |
| **R-01.5** Режим взаимодействия с субстратом | Синхронный блокирующий ←→ Асинхронный событийный | `Brain`, `Tools` |

#### Архитектурные напряжения

1. **Универсальность vs. Специализация Brain:** Единый семантический процессор обеспечивает гомогенность, но может ограничивать эффективность. Федерация процессоров повышает эффективность, но усложняет интеграцию и может нарушить семантическую гомогенность.

2. **Богатство модальностей vs. Сложность интеграции:** Расширение набора Tools увеличивает возможности, но создаёт нагрузку на AfferentialSynthesis (AD-05) для интеграции мультимодального контекста.

#### Влияние параметров AI-P

- При `Tools = Absent` (A–B4): домен сводится к выбору характеристик Brain. Моторные интерфейсы не проектируются.
- При `Tools = Present` (C–F): требуется проектирование сенсорно-моторных интерфейсов и их интеграции с Brain.

### AD-02: Когнитивное управление и временной режим

#### Типы архитектурных решений

| Решение | Диапазон вариативности | Управляющие параметры |
|---------|----------------------|----------------------|
| **R-02.1** Модель управления активацией | Детерминированная ←→ Событийная ←→ Стохастическая ←→ Адаптивная | `control_type` |
| **R-02.2** Гранулярность дискретизации времени | Отсутствие часов ←→ Крупные такты ←→ Тонкая дискретизация | `CognitiveClock` |
| **R-02.3** Модель синхронизации | Барьерная (ожидание группы) ←→ Асинхронная (независимые акты) | `SynchronizationMode` |
| **R-02.4** Содержание ClockState | Минимальное (список ФЯ) ←→ Расширенное (статусы, метрики, история) | `ClockState` |
| **R-02.5** Режим доступа к ClockState | None ←→ Direct ←→ Policy ←→ Both | `AfferentialPolicy.ClockState` |
| **R-02.6** Модель параллелизма | Строго последовательное ←→ Ограниченный параллелизм ←→ Полный параллелизм | `SynchronizationMode`, `CC` |

#### Архитектурные напряжения

1. **Предсказуемость vs. Адаптивность:** Детерминированное управление обеспечивает воспроизводимость, но не позволяет адаптировать стратегию. Адаптивное управление гибко, но требует процедурной памяти и может порождать нестабильность.

2. **Параллелизм vs. Согласованность:** Асинхронный режим увеличивает пропускную способность, но создаёт проблему согласованности ClockState между параллельно работающими ФЯ.

3. **Глубина ClockState vs. Когнитивная нагрузка:** Расширенный ClockState даёт больше контекста, но увеличивает объём обстановочной афферентации.

#### Влияние параметров AI-P

- При `CognitiveClock = Absent` (A): нет дискретизации, нет ClockState.
- При `AfferentialPolicy.ClockState = Policy` (B2–B3): требуется SE для хранения истории; PMA для проекции.
- При `control_type = Адаптивный` (E–F): CC сам становится объектом эволюции (AD-10); требуется `procedural` память.

### AD-03: Коннектом и топология поведенческих переходов

#### Типы архитектурных решений

| Решение | Диапазон вариативности | Управляющие параметры |
|---------|----------------------|----------------------|
| **R-03.1** Тип связности | Только явные ←→ Только неявные ←→ Гибридные | `Connectome.type` |
| **R-03.2** Архитектура явных связей | Статический граф ←→ Таблица переходов ←→ Правила ←→ Динамический граф | `Connectome.type`, `CorrectionObjects.Connectome_Explicit` |
| **R-03.3** Природа неявных связей | Через общую память ←→ Через специализированные каналы в опыте | `Connectome.type`, `SubjectiveExperience` |
| **R-03.4** Динамика топологии | Статическая ←→ Модифицируемая вручную ←→ Эволюционируемая автономно | `CorrectionObjects.Connectome_*` |

#### Архитектурные напряжения

1. **Детерминизм vs. Гибкость:** Явные связи гарантируют предсказуемость, но ограничивают адаптивность. Неявные связи обеспечивают гибкость, но затрудняют предсказание траекторий.

2. **Прозрачность vs. Богатство:** Explicit-коннектом легко анализировать. Implicit-коннектом богаче, но действие одной ФЯ на другую опосредовано неявно через SE.

3. **Стабильность vs. Пластичность:** Статический коннектом стабилен. Эволюционируемый адаптивен, но может деградировать при ошибках обратной афферентации.

#### Влияние параметров AI-P

- При `type = Explicit` (A–B1): пространство сужается до проектирования графа переходов.
- При `type = Implicit` (B2–D): основной вопрос — как SE опосредует влияние ФЯ друг на друга.
- При `type = Both` (E–F): необходимо проектировать оба типа связей и их взаимодействие.

### AD-04: Функциональная ячейка и структура когнитивного акта

#### Типы архитектурных решений

| Решение | Диапазон вариативности | Управляющие параметры |
|---------|----------------------|----------------------|
| **R-04.1** Формат Intent | Фиксированная строка ←→ Структурированный объект ←→ Параметризованная цель | `Intent` |
| **R-04.2** Конфигурация AfferentialPolicy | Отсутствие ←→ Минимальная (только Direct) ←→ Полная (4 фактора) | `AfferentialPolicy.*` |
| **R-04.3** Конфигурация SubjectivityPolicy | Отсутствие ←→ Минимальная (ClockState) ←→ Полная (Frame + Experience) | `SubjectivityPolicy.*` |
| **R-04.4** Характер ResultAcceptor ФЯ | Отсутствие ←→ Структурный (фильтр вывода) | `ResultAcceptor` |
| **R-04.5** Степень пластичности политик | Фиксированные ←→ Модифицируемые эволюцией | `CorrectionObjects.FC_Components` |
| **R-04.6** Гранулярность ФЯ | Крупные (сложные многошаговые акты) ←→ Мелкие (атомарные операции) | `FC` |
| **R-04.7** Парадигма валидации результата | Детерминированная структурная проверка ←→ Семантическая оценочная валидация | `ResultAcceptor` |

#### Архитектурные напряжения

1. **Богатство контекста vs. Стоимость синтеза:** Полная AfferentialPolicy обеспечивает глубокий контекст, но требует развитого AfferentialSynthesis и PMA.

2. **Гранулярность ФЯ vs. Управляемость:** Мелкие ФЯ дают точный контроль, но требуют сложного CC и большого коннектома. Крупные ФЯ проще управлять, но теряется атомарность.

3. **Пластичность vs. Стабильность политик:** Модифицируемые политики позволяют эволюцию, но могут деградировать. Фиксированные стабильны, но не адаптируются.

4. **Детерминированная vs. Семантическая валидация:** Структурная проверка результата проста и предсказуема, но не может оценить семантическое качество. Семантическая оценка (brain-as-a-judge) мощнее, но дорога и менее детерминирована.

#### Влияние параметров AI-P

- При `AfferentialPolicy = Absent` (A): ФЯ реактивна — нет контекста, нет субъектности.
- При `SubjectivityPolicy.Attribution = ClockState` (B2): минимальная атрибуция.
- При `SubjectivityPolicy.Attribution = Frame` (B3–F): полная атрибуция.
- При `CorrectionObjects.FC_Components = affected` (D–F): политики ФЯ становятся объектом эволюции.

### AD-05: Афферентный контекст и формирование кадра сознания

#### Типы архитектурных решений

| Решение | Диапазон вариативности | Управляющие параметры |
|---------|----------------------|----------------------|
| **R-05.1** Набор активных факторов | Только мотивация+пуск ←→ +Ситуация ←→ +Память ←→ Все 4 | `AfferentialPolicy.*` |
| **R-05.2** Механизм интеграции факторов | Конкатенация ←→ Взвешенное слияние ←→ Структурная композиция | `AfferentialSynthesis` |
| **R-05.3** Стратегия формирования Frame | Фиксированный шаблон ←→ Динамический по AfferentialPolicy | `AfferentialPolicy`, `PMA` |
| **R-05.4** Глубина привлечения памяти | Нет ←→ Только эпизодическая ←→ +Прикладная ←→ +Процедурная | `PMA.SelectionDepth` |
| **R-05.5** Управление релевантностью | Неуправляемая (всё доступное) ←→ Управляемая фокусировка | `PMA`, `AfferentialPolicy` |
| **R-05.6** Стратегия загрузки контекста | Жадная предзагрузка (eager pre-fetch) ←→ Ленивый добор по запросу (lazy on-demand) | `AfferentialSynthesis`, `PMA` |

#### Архитектурные напряжения

1. **Глубина контекста vs. Латентность:** Привлечение глубоких слоёв памяти обогащает контекст, но увеличивает время формирования Frame.

2. **Релевантность vs. Полнота:** Строгая фокусировка PMA повышает релевантность, но может отсечь важную информацию.

3. **Единообразие vs. Специализация синтеза:** Единый механизм синтеза прост, но может быть неоптимальным для разных типов ФЯ.

4. **Жадная vs. Ленивая загрузка:** Предзагрузка снижает латентность повторных обращений, но увеличивает стоимость первоначальной сборки контекста; ленивый добор экономичнее, но может вызывать задержки при первом обращении к слою памяти.

#### Влияние параметров AI-P

- При `AfferentialSynthesis = Absent` (A): синтез не проектируется; ФЯ работает без контекста.
- При `PMA = Absent` (B1): синтез ограничен текущим состоянием.
- При `PMA.SelectionDepth = Applied` (C–D): синтез привлекает знания.
- При `PMA.SelectionDepth = Procedural` (E–F): синтез привлекает навыки.

### AD-06: Эфферентная реализация и интерфейс действия

#### Типы архитектурных решений

| Решение | Диапазон вариативности | Управляющие параметры |
|---------|----------------------|----------------------|
| **R-06.1** Стратегия таргетирования субстрата | Brain only ←→ Tools only ←→ Смешанная маршрутизация (Brain+Tools) | `EfferentialPolicy.TargetSubstrate` |
| **R-06.2** Контекстная насыщенность эфферентного акта | Без контекста (None) ←→ ClockState ←→ Frame ←→ Полный контекст (Both) | `EfferentialPolicy.SubjectiveContext` |
| **R-06.3** Разделение внутренних и внешних актов | Единый (слитный) контур ←→ Раздельные контуры с разной логикой | `Tools`, `TargetSubstrate` |
| **R-06.4** Трассировка результата акта | Минимальная фиксация ←→ Расширенная с разделением Brain_Output / Tools_Output | `SubjectivityPolicy.Experience` |

#### Архитектурные напряжения

1. **Контекстная насыщенность vs. Скорость эфферентации:** Передача полного контекста (Both) повышает качество действия, но увеличивает латентность и стоимость вызова.

2. **Единый vs. Раздельный контур:** Слитный контур прост, но не позволяет дифференцировать логику для когнитивных и инструментальных актов. Раздельные контуры гибче, но усложняют архитектуру.

3. **Полнота трассировки vs. Компактность опыта:** Расширенная трассировка обеспечивает богатую обратную связь для эволюции, но увеличивает объём записываемого опыта.

#### Влияние параметров AI-P

- При `Tools = Absent` (A–B4): только Brain-directed акты; AD-06 сводится к выбору контекстной насыщенности.
- При `SubjectiveContext = None` (A): эфферентный акт не содержит субъектного контекста — чистая реактивность.
- При `SubjectiveContext = Both` (B4–F): максимальная контекстная насыщенность.
- При `Tools = Present` (C–F): необходимо проектировать разделение внутренних/внешних актов и стратегию маршрутизации.

### AD-07: Субъективный опыт и архитектура памяти

#### Типы архитектурных решений

| Решение | Диапазон вариативности | Управляющие параметры |
|---------|----------------------|----------------------|
| **R-07.1** Формат единицы опыта (SEU) | Минимальный (ClockState) ←→ Полный (ClockState+Frame+Intent+Result+Timestamp) | `SubjectivityPolicy.*` |
| **R-07.2** Глубина эпизодической памяти | Отсутствие ←→ Recency-only (окно) ←→ Unlimited (полная история) | `episodic.Depth` |
| **R-07.3** Архитектура прикладного слоя | Отсутствие ←→ Сеть знаний предметной области | `applied.status` |
| **R-07.4** Архитектура процедурного слоя | Отсутствие ←→ Сеть когнитивных схем (навыки) | `procedural.status` |
| **R-07.5** Архитектура ценностного слоя | Отсутствие ←→ Экстрагированные ценностные ориентиры | `values.status` |
| **R-07.6** Модель хранения и индексации | Линейный поток ←→ Индексированное хранилище ←→ Граф знаний | SE, MemoryLayers |
| **R-07.7** Модель персистентности опыта | State Machine (перезапись текущего состояния) ←→ Event Sourcing (append-only журнал событий) | SE, MemoryLayers |

#### Архитектурные напряжения

1. **Полнота записи vs. Масштабируемость:** Unlimited эпизодическая память сохраняет всю историю, но создаёт проблему масштабируемости хранения и поиска.

2. **Иерархическая глубина vs. Сложность:** Добавление каждого нового слоя памяти увеличивает глубину субъектности, но требует соответствующих процессов консолидации.

3. **Структурированность vs. Гибкость:** Строго структурированная память обеспечивает эффективный доступ, но ограничивает типы хранимого опыта.

4. **Event Sourcing vs. State Machine:** Event Sourcing обеспечивает полную аудируемость и возможность «перепроигрывания» истории, но растёт линейно. State Machine компактен, но теряет историю.

#### Влияние параметров AI-P

- При `SE = Absent` (A): домен неактивен.
- При `episodic.Depth = Recency-only` (B1): только скользящее окно.
- При `applied = Present` (C–F): появляется слой обобщённых знаний.
- При `procedural = Present` (E–F): появляется слой навыков.

### AD-08: Проекция и присвоение опыта

#### Типы архитектурных решений

| Решение | Диапазон вариативности | Управляющие параметры |
|---------|----------------------|----------------------|
| **R-08.1** Механизм фокусировки PMA | Без фокусировки ←→ Управляемая AfferentialPolicy ←→ Адаптивная фокусировка | `PMA`, `AfferentialPolicy` |
| **R-08.2** Механизм трансформации в Frame | Прямое копирование ←→ Семантическая компрессия ←→ Генеративная проекция | `PMA`, `Frame` |
| **R-08.3** Объём Frame | Минимальный (ключевые факты) ←→ Расширенный (контекст) ←→ Максимальный (полная проекция) | `Frame`, `PMA.SelectionDepth` |
| **R-08.4** Полнота атрибуции | Только ClockState ←→ ClockState + Frame | `SubjectivityPolicy.Attribution` |
| **R-08.5** Содержание записи опыта | Только Attribution ←→ +Brain_Output ←→ +Tools_Output ←→ ALL | `SubjectivityPolicy.Experience` |
| **R-08.6** Механизм записи (EgoAfferentation) | Синхронная (блокирующая) ←→ Асинхронная (фоновая) | `EgoAfferentation` |

#### Архитектурные напряжения

1. **Объём сознания vs. Вычислительная стоимость:** Максимальный Frame обеспечивает глубокое осознание, но дорог в формировании и передаче.

2. **Проекция vs. Присвоение (PMA vs. EgoAfferentation):** Эти два процесса образуют операторную пару. Несбалансированность приводит к деградации сознания как динамического инварианта.

3. **Уникальность vs. Согласованность Frame:** Каждая ФЯ получает уникальный Frame. Это обеспечивает специализацию, но создаёт проблему согласованности «картины мира» между параллельно работающими ФЯ.

#### Влияние параметров AI-P

- При `PMA = Absent` (A–B1): проекция опыта не проектируется.
- При `Attribution = ClockState` (B2): минимальная субъективность.
- При `Attribution = ClockState+Frame` (B3–F): полная субъективность.
- При `SelectionDepth = Procedural` (E–F): проекция включает навыки.

### AD-09: Консолидация опыта

#### Типы архитектурных решений

| Решение | Диапазон вариативности | Управляющие параметры |
|---------|----------------------|----------------------|
| **R-09.1** Временная консолидация | Отсутствие ←→ Скользящее окно ←→ Иерархические эпизоды | `EpisodicTimeConsolidation` |
| **R-09.2** Прикладная консолидация | Отсутствие ←→ Извлечение паттернов ←→ Построение онтологий | `AppliedConsolidation` |
| **R-09.3** Процедурная консолидация | Отсутствие ←→ Формирование когнитивных схем ←→ Мета-навыки | `ProceduralConsolidation` |
| **R-09.4** Триггеры консолидации | Периодические ←→ Событийные ←→ Пороговые | `ConsolidationPolicy` |
| **R-09.5** Пластичность политик консолидации | Фиксированные ←→ Модифицируемые эволюцией | `CorrectionObjects.SubjExp_ConsolidationPolicy` |

#### Архитектурные напряжения

1. **Скорость обобщения vs. Сохранение деталей:** Агрессивная консолидация быстро формирует знания, но теряет нюансы. Консервативная сохраняет детали, но замедляет обучение.

2. **Автономность vs. Управляемость:** Автоматическая консолидация обеспечивает автономность, но может формировать ошибочные обобщения.

3. **Согласованность слоёв vs. Независимость:** Слои памяти связаны, но должны оставаться достаточно независимыми.

#### Влияние параметров AI-P

- При `Consolidation = Absent` (A–B4): домен неактивен.
- При `AppliedConsolidation = Present` (C–F): субъект строит модель мира.
- При `ProceduralConsolidation = Present` (E–F): субъект формирует навыки.
- При `ConsolidationPolicy = affected` (E–F): субъект «учится учиться».

### AD-10: Адаптация и эволюция

#### Типы архитектурных решений

| Решение | Диапазон вариативности | Управляющие параметры |
|---------|----------------------|----------------------|
| **R-10.1** Режим эволюции | Отсутствие ←→ Manual (human-in-the-loop) ←→ Auto (автономная) ←→ Full (рефлексивная) | `FeedbackAfferentation.status` |
| **R-10.2** Архитектура Функциональных Систем | Отсутствие ←→ Простые (1:1) ←→ Конкурентные (1:m) | `FS.Concurrency` |
| **R-10.3** Источник эталонов | Отсутствие ←→ Внешний (оператор) ←→ Внутренний (субъект) | `ResultAcceptor.ResultModel` |
| **R-10.4** Механизм сравнения | Отсутствие ←→ Внешний ←→ Внутренний | `ResultAcceptor.ComparisonMechanism` |
| **R-10.5** Состав объектов коррекции | Минимальный (FC) ←→ Средний (+Connectome) ←→ Расширенный (+ConsolidationPolicy) ←→ Полный (+CC, +FS_Acceptors) | `CorrectionObjects.*` |
| **R-10.6** Политика разрешения конфликтов | Отсутствие ←→ Приоритетная ←→ Голосование ←→ Арбитражная | `ConflictResolutionPolicy` |
| **R-10.7** Глубина мутации | Параметрическая адаптация (веса, пороги, промпты) ←→ Топологическая модификация (добавление/удаление ФЯ и связей) | `CorrectionObjects.*`, `FeedbackAfferentation` |
| **R-10.8** Глубина рефлексивности | Модификация структур ←→ Модификация механизмов модификации | `CorrectionObjects.CC_Strategies`, `CorrectionObjects.FS_Acceptors` |

#### Архитектурные напряжения

1. **Автономность vs. Контролируемость:** Автономная эволюция обеспечивает самодостаточность, но создаёт риск неуправляемых изменений.

2. **Широта коррекции vs. Стабильность:** Расширение объектов коррекции увеличивает адаптивность, но каждый новый объект — потенциальная точка дестабилизации.

3. **Конкуренция ФС vs. Целостность субъекта:** При 1:m одна ФЯ получает конкурентные сигналы от нескольких ФС. Необходима политика разрешения конфликтов.

4. **Внутренние vs. Внешние эталоны:** Внутренние обеспечивают автономность, но могут «зациклиться». Внешние обеспечивают объективность, но создают зависимость.

5. **Параметрическая vs. Топологическая мутация:** Параметрическая адаптация безопаснее и обратимее. Топологическая мощнее, но рискованнее — добавление/удаление ФЯ может нарушить инварианты коннектома.

#### Влияние параметров AI-P

- При `L4 = Absent` (A–C): домен неактивен.
- При `FeedbackAfferentation = Manual` (D): все изменения инициируются извне.
- При `FeedbackAfferentation = Self` (E): автономная коррекция.
- При `Concurrency = 1:m` (F): необходима ConflictResolutionPolicy.
- При `CorrectionObjects.CC_Strategies = affected` (F): CC становится объектом эволюции.

### AD-11: Структурный субстрат и морфология

#### Типы архитектурных решений

| Решение | Диапазон вариативности | Управляющие параметры |
|---------|----------------------|----------------------|
| **R-11.1** Формат представления структурных элементов | Фиксированная схема ←→ Декларативная конфигурация ←→ Исполняемый код | `SubjectMorphology` |
| **R-11.2** Модель исполнения | Интерпретация ←→ Компиляция ←→ Гибрид | `SubjectMorphology` |
| **R-11.3** Поддержка модификации в runtime | Отсутствие ←→ Частичная (hot-reload политик) ←→ Полная (hot-reload структуры) | `SubjectMorphology`, `CorrectionObjects.*` |
| **R-11.4** Разграничение Brain и Morphology | Слитые ←→ Раздельные с интерфейсом | `Brain`, `SubjectMorphology` |
| **R-11.5** Обеспечение целостности при модификации | Без гарантий ←→ Транзакционная ←→ Версионированная (git-like) | `SubjectMorphology` |

#### Архитектурные напряжения

1. **Гибкость модификации vs. Целостность:** Полный hot-reload обеспечивает гибкость эволюции, но создаёт риски нарушения целостности при параллельных модификациях.

2. **Семантическая гомогенность vs. Разделение ответственности:** Аксиома семантической гомогенности требует, чтобы Brain мог оперировать структурами Morphology. Но архитектурно Brain и Morphology выполняют разные роли.

3. **Эффективность vs. Универсальность формата:** Специализированный формат эффективен, но ограничивает типы возможных ФЯ и политик. Универсальный гибок, но менее эффективен.

#### Влияние параметров AI-P

- Morphology присутствует всегда (Carrier).
- При `L4 = Absent` (A–C): Morphology — пассивный носитель.
- При `L4 = Present` (D–F): Morphology — активный субстрат модификации.
- Версионированная целостность (git-like) особенно актуальна для F, где конкурентные ФС могут генерировать противоречивые модификации.
